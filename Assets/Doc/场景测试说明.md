# 场景测试说明 (Assets/Scenes)

> 本文记录 `Assets/Scenes/` 下每个 `*.unity` 场景的测试意图,以及为覆盖 v2 架构关键能力(多纹理分批、8-slot 着色器路径、性能基线对比)而**计划新增**的场景。
> 关联文档:[架构设计-v2.md](架构设计-v2.md) · [实施计划.md](实施计划.md) · [测试用例.md](测试用例.md) · [实施状态审计.md](实施状态审计.md)。

约定:
- "状态" 列: ✅ 场景已存在且与脚本对齐 / 🟡 存在但功能与 v2 不完全对齐 / 📋 设计已定,场景待创建。
- 所有"待创建"场景**不要用命令行写 `.unity` 文件**,需在 Unity Editor 中搭建并保留 `.meta` GUID 后入库。

### 1.1 现有场景缺口

- **多纹理 (>3) 触发 `MergeBatcher` 分批**: 现有 #5/#10 都只 1~2 个纹理,从未真正逼近 `MaxImageSlots=3` 上限,**无法验证 `Plan_SplitsWhenOver` (TC-MB-02) 在真实场景里的行为**。
- **`HUD_8_TEX_SLOTS` 着色器分支**: 既无场景启用 `UIPrefabManager.Enable8TexSlots(material)`,也无对应 4..7 纹理装载的资源。**TASK-18 在场景层完全无演示**。
- **`PerfProbe.Flush` CSV 端到端**: 现有 #11/#8 仅手动观测,无场景跑完一段后自动 `Flush` 落盘并显示 CSV 路径 → 与 [T-D-02](实施状态审计.md) 同源。
- **`UIMeshData` vs `UIMeshDataX` 同输入下的对比**: 单元层 [TC-MX-01..06](测试用例.md) 已覆盖代数等价,但**无场景级 A/B 帧时间对比**;`UseSlim(bool)` 在 #10 中存在但无并排同帧对照。

## 2. 现有场景: 详细文档卡片

> 每张卡片格式: 简介 / 测试目标 / 操作步骤 / 预期结果 / 对应测试用例 / 关联 v2 模块。

### 2.1 SingleTextRenderer.unity

- **简介**: 一个 Canvas + 一个 UGUI `Text`,无自定义脚本。
- **测试目标**: 提供 v2 管线参照组的**最小 UGUI 基线**(BaseLine-A)。
- **操作步骤**: Play → 观察 Profiler Frame Debugger 的 DrawCall 数与 Batch 个数。
- **预期结果**: DrawCall = 1 (Canvas 自带);CPU UI.Geometry 阶段 ~0.0x ms。
- **对应测试用例**: 无单元测试。Performance Test 在 §3 T-PB-01 中将基于此场景采样基线。
- **关联模块**: 无 v2 模块直接相关,纯对照组。

### 2.2 BatchTextRenderer.unity

- **简介**: 1 Image + 1 Text,通过 [BatchUIDataRender.cs](../Scripts/HOT/BatchUIDataRender.cs) 抽取网格手动合并。
- **测试目标**: 验证**最简合批通路**(无 MergeBatcher) 与原 v1 行为一致。
- **操作步骤**: Play → 点击 `ReCreate` 按钮 → 检查 `compbine_mesh` 顶点数 = Image.4 + Text.N。
- **预期结果**: 1 个 `Graphics.DrawMesh` 调用;无 LogError;材质纹理槽 0 = font, 槽 1 = image。
- **对应测试用例**: 与 [TC-UM-01..03](测试用例.md) 同语义路径的场景验证。
- **关联模块**: `UIMeshData` + `VertexHelperUtils.FillData2`。
- **已知差距**: 未走 `MergeBatcher`,纹理超 3 时无分批保护。

### 2.3 TestTextGraphics.unity

- **简介**: 单个 UGUI Text + `TestTextClick` 监听 `IPointerClickHandler`。
- **测试目标**: 验证字体纹理通过 `RawImage` 可视化、点击命中文本顶点子区域。
- **操作步骤**: Play → 点击 Text → 在 `RawImage` 上看到子区域被点亮(原脚本逻辑)。
- **预期结果**: `vertexs` 列表为命中的 UIVertex;`textureSize` 与 `fontTexture` 尺寸一致。
- **对应测试用例**: 无;为字体纹理子区域功能研究,不在 v2 关键路径。
- **关联模块**: 仅 UGUI 内部 API。

### 2.4 TestUITextColor.unity

- **简介**: 多 Text,设置不同 `Color`。
- **测试目标**: 验证 `Color32` 在 `FillVertex → FillToDrawData` / `FillWithMatrix` 中不丢失、不截断、不被 `default` 替换为白色(当原色非默认时)。
- **操作步骤**: Play → 切换不同颜色 Text → 截图比对实际颜色。
- **预期结果**: 屏幕颜色 = 编辑器设置颜色;非黑色 Text 不出现整体变白。
- **对应测试用例**: [TC-UM-02 / TC-MX-02](测试用例.md) 的视觉回归。
- **关联模块**: `UIMeshData.FillToDrawData` 中 `default Color32 → white` 的兜底分支。

### 2.5 TestGraphicsData.unity

- **简介**: Simple Image + 9-Slice Image + Text 共存。

## 3. 计划新增场景 (📋 待在 Unity Editor 中搭建)

> 新增场景按 ID 编号,后续可作为 [实施状态审计.md](实施状态审计.md) 的 T-S-A-* 系列工作项被引用。
> 所有新场景应放在 `Assets/Scenes/`,命名规则 `Scene_<域>_<编号>_<标识>.unity`。
> 共享脚手架已补齐: [PerfBaselineRunner.cs](../Scripts/HOT/PerfBaselineRunner.cs) 负责自动采样/Flush CSV, [PrefabBatchSceneRunner.cs](../Scripts/HOT/PrefabBatchSceneRunner.cs) 负责批量 holder、`useSlim`、`enable8TexSlots` 与 `HolderBatchRenderer` 主路径。
> 若要**保留 BatchTextRenderer 的手动操作方式**(Inspector 按钮触发 `ReCreate` / 手动改文案 / 手动换图),优先使用 [BatchMergeBatcherRender.cs](../Scripts/HOT/BatchMergeBatcherRender.cs)；它直接以场景中的 `UIPrefabOwner` 列表为输入,但内部走 `MergeBatcher` 正式渲染路径。

### 3.1 域: 多纹理 (Texture Slot)

#### 3.1.0 推荐搭建方式：保留 BatchTextRenderer 风格的手动场景

适用于 `T-S-A-01..04`：

1. 新建空场景或复制 `BatchTextRenderer.unity` 的 **Canvas / Camera / EventSystem / 按钮布局**。
2. **不要复用** `BatchUIDataRender.cs`；删除旧控制脚本。
3. 在 Canvas 下手工摆放若干 `UIPrefabOwner` 物体（推荐直接拖入 `Resources/UI/UIName.prefab`、`Resources/UI/UITitle.prefab` 作为起点）。
4. 新建空物体 `BatchMergeBatcher`，挂载 [BatchMergeBatcherRender.cs](../Scripts/HOT/BatchMergeBatcherRender.cs)。
5. 在 Inspector 里绑定：
   - `font`
   - `uiCamera`
   - `uiRoot`
   - `statusText`（可选）
   - `owners`（把场景中的 `UIPrefabOwner` 顺序拖进去）**或** `ownerPrefabs`（若想运行时独立创建测试对象）
6. 推荐开关：
   - `displayMode = UIAndDataRender`：源 UGUI 与 `Graphics.DrawMesh` 合批结果同时显示，适合叠加比对
   - `displayMode = UIOnly`：只显示源 UGUI，`BatchMergeBatcherRender` 仍可重建数据，但 `LateUpdate` 不绘制合批 mesh
   - `displayMode = DataRenderOnly`：隐藏源 UGUI，只显示 `Graphics.DrawMesh` 结果；隐藏时会把源对象 `CanvasGroup.alpha` 设为 0，并临时关闭 `interactable / blocksRaycasts`
   - `displayMode` 支持 PlayMode 实时切换：可直接在 Inspector 改 enum，不需要重新 `ReCreate()`
   - `instantiateOwnersAtRuntime = true`：不直接使用场景里的源对象，而是在运行时从 `ownerPrefabs` 独立创建一批测试对象
   - `runtimeSourceRoot`：可选，给运行时创建的源对象一个单独容器
   - `runtimeStartPosition / runtimeStep`：控制运行时测试对象的排布
7. 手动操作沿用旧习惯：
   - `ReCreate()`：重新注册 + 重新合批
   - `ApplyTextOverride()`：改指定 holder 的指定 draw text
   - `ApplySpriteOverride()`：改指定 holder 的指定 draw sprite
   - `CycleSprite()`：循环切图，适合 `T-S-A-04`
   - `FlushProbe()`：导出 CSV
   - `StartEightSlotComparison()`：若同物体或 `comparisonRunner` 绑定了 `EightSlotPerfRunner`，可一键启动 3-slot / 8-slot 对比采样
8. 导出的 CSV 文件名会保留关键上下文（例如 `3slot/8slot`、`managed/slim`、`ownersN`、`batchesN`、`displayMode`）；文件内容的 `context,value` 段也会写入 `useSlim` 等关键参数，避免后续整理样本时信息丢失。

这样保留了 `BatchTextRenderer` 的“手工按钮驱动”体验，但渲染路径已经切到 `UIPrefabManager` → `MergeBatcher` → `HolderBatchRenderer`。

- **预期结果**:
  - 启用前: 触发 `MergeBatcher` 分批告警 (6 > 3)
  - 启用后: `MaxImageSlots == 7`;`batches.Count == 1`;**屏幕显示 6 张贴图正确,且无告警**
  - Frame Debugger 显示 `HUD_8_TEX_SLOTS` 关键字为 enabled,采样 `_MainTex4 / _MainTex5`
- **对应测试用例**: [TC-TST-07 / TC-TST-08](测试用例.md) 的场景级集成验证。
- **关联模块**: `TextureSlotTable.ExpandTo(7)` + `MaterialBinder` + Shader `UIE-AtlasBlit.shader` 的 `HUD_8_TEX_SLOTS` 分支。
- **资源需求**: 6 张不同贴图。
- **附加产物**:
  - `EightSlotPerfRunner` 输出 `*_3slot` / `*_8slot` 两份 CSV,文件名还会携带槽位模式与批次上下文,用于记录 CPU fill_ms / draw_calls 的 A/B 数据
  - 最近一次 Editor 本地采样结果:
    - `useSlim = false`: `3-slot fill_ms avg 0.254 / draw_calls 2` → `8-slot fill_ms avg 0.235 / draw_calls 1`
    - `useSlim = true`: `3-slot fill_ms avg 1.218 / draw_calls 2` → `8-slot fill_ms avg 0.637 / draw_calls 1`
  - `RenderDoc` 当前环境不可用,因此 GPU 侧先以 Profiler Detailed 为主;待帧捕获工具恢复后再补 shader 指令/采样器压力证据

#### 📋 T-S-A-04 · Scene_Tex_04_DynamicSwap.unity

- **简介**: 4 个 `UIImage`,运行时按按钮**循环替换** sprite (`SetSprite(index, ...)`),触发 `TextureSlotTable.Register` / `Unregister` 的 Swap-with-Last 路径。
- **推荐控制脚本**: [BatchMergeBatcherRender.cs](../Scripts/HOT/BatchMergeBatcherRender.cs)（使用 `CycleSprite()`）
- **测试目标**: 验证动态换图时:
  - `HolderNotifier` 按反向索引正确通知到对应 mesh
  - `UIMeshDataX.UpdateTextureIndex` 正确改写 `uv.z`
  - 无 mesh 残留旧槽号
- **预期结果**: 每次换图后屏幕显示新贴图;`PerfProbe.Fill` marker 显示通知耗时 O(受影响数) 而非 O(全 holder)。
- **对应测试用例**: [TC-HN-01 / TC-HN-02 / TC-TST-04 / TC-TST-05](测试用例.md) 的场景集成。
- **关联模块**: `TextureSlotTable.Unregister` Swap-with-Last + `HolderNotifier.ReplaceTextureID / RemoveTextureID`。

### 3.2 域: 性能基线 (Performance Baseline)

> 目标: 建立**5 个对照场景**的统一基线矩阵,使后续优化(Burst SoA、8-slot 等)可以做横向对比。
> 所有 PB 场景应**自动 `PerfProbe.Flush` CSV** 到项目内临时目录 `Logs/PerfProbe/`（Player 环境回退到 `Application.persistentDataPath`）,并在 UI 上显示 CSV 路径。

#### 📋 T-PB-01 · Scene_Perf_01_UGUI_Baseline.unity

- **简介**: N 个 (默认 N=100) 原生 UGUI Image+Text 组合 prefab,无任何 v2 自定义渲染。
- **测试目标**: **基线 A (UGUI 原生)**。
- **关键 metric**: `Canvas.SendWillRenderCanvases` 耗时 / DrawCall 数 / SetPass 数。
- **附加脚本**: 新增 [PerfBaselineRunner.cs](../Scripts/HOT/PerfBaselineRunner.cs) MonoBehaviour,Play 后跑 600 帧自动 `PerfProbe.Flush("ugui_n100")`。
- **对应测试用例**: 无单元 TC;作为后续 Performance Test 的输入基线。

#### 📋 T-PB-02 · Scene_Perf_02_UIMeshData_Baseline.unity

- **简介**: 同 T-PB-01 的 N,但全部 holder 走 `DataPrefabHolder<UIMeshData>` (per-instance 数组)。`UseSlim(false)`。
- **测试目标**: **基线 B (v2 独立模式)**。
- **预期对比**: vs T-PB-01,GC Alloc 应显著减少;DrawCall 应降至 ~1(单纹理情况)。
- **附加脚本**: 复用 [PerfBaselineRunner.cs](../Scripts/HOT/PerfBaselineRunner.cs) + [PrefabBatchSceneRunner.cs](../Scripts/HOT/PrefabBatchSceneRunner.cs),tag 改为 `"meshdata_n100"`。

#### 📋 T-PB-03 · Scene_Perf_03_UIMeshDataX_Baseline.unity

- **简介**: 同 T-PB-02,但 `UseSlim(true)` → `DataPrefabHolder<UIMeshDataX>`(共享 `UIGeometry` 池)。
- **测试目标**: **基线 C (v2 共享模式)**。
- **预期对比**: vs T-PB-02,长期跑后 GC Alloc 应进一步下降(无 per-instance 数组分配)。
- **关联**: [TC-LIFE-01](测试用例.md) 切换路径的稳态性能侧面。
- **附加脚本**: 复用 [PerfBaselineRunner.cs](../Scripts/HOT/PerfBaselineRunner.cs) + [PrefabBatchSceneRunner.cs](../Scripts/HOT/PrefabBatchSceneRunner.cs),其中 `useSlim = true`。

#### 📋 T-PB-04 · Scene_Perf_04_UIMeshDataX_WithJob.unity

- **简介**: 同 T-PB-03 + `useJob = true` 走 `ManagedCodeInJob` 桥接。
- **测试目标**: **基线 D (v2 共享模式 + Job 桥接)**。
- **预期对比**: vs T-PB-03,合并阶段移到 worker 线程,主线程 `UIData.MergeJob` marker 时间应下降。
- **附加产物**: 为后续 [T-A-04 Burst SoA](实施状态审计.md) 提供 vs 优化前 的基线。

#### 📋 T-PB-05 · Scene_Perf_05_HighCount_Stress.unity

- **简介**: T-PB-03 配置,但 `count = 1000`,加 `changePos = true` 模拟战斗 HUD 每帧抖动场景。
- **测试目标**: **压力基线**,验证 GeometryBuffer 扩容、FreeListAllocator Release 5 case 路径在长时高负载下无泄漏。
- **预期结果**:
  - 600 帧后 `geometry.openVertexList.Count` 收敛(不无限增长)
  - `PerfProbe.AvgMs` < 目标设备 16ms 帧预算的某个 budget(具体数字在跑出后写入 [实施计划.md](实施计划.md) 验收标准)
  - `LogAssert.NoUnexpectedReceived()` 无未处理告警(超容量除外)
- **对应测试用例**: [TC-UG-* (Release case 0..4)](测试用例.md) 与 [TC-GB-01 (EnsureVertexCapacity 增长)](测试用例.md) 的长稳验证。
